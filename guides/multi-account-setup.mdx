The Kiro Auth plugin supports managing multiple AWS accounts simultaneously, with intelligent account rotation to maximize your usage quotas and minimize rate limiting.

## How Multi-Account Works

The plugin maintains a pool of authenticated accounts and automatically selects the best account for each request based on your configured strategy.

<Info>
Accounts are stored in a local SQLite database at `~/.config/opencode/kiro.db` (or `%APPDATA%\opencode\kiro.db` on Windows).
</Info>

## Account Selection Strategies

### Sticky (Default)

<CodeGroup>
```json opencode.json
{
  "account_selection_strategy": "sticky"
}
```

```json kiro.json
{
  "account_selection_strategy": "sticky"
}
```
</CodeGroup>

Sticks to the same account until it becomes unavailable (rate-limited or unhealthy). This is the simplest strategy and provides consistent behavior.

**Best for**: Single-account usage or when you want predictable account selection.

### Round-Robin

```json
{
  "account_selection_strategy": "round-robin"
}
```

Cycles through all available accounts in order. Each request uses the next account in the rotation.

**Best for**: Evenly distributing load across multiple accounts.

### Lowest-Usage (Recommended)

```json
{
  "account_selection_strategy": "lowest-usage"
}
```

Always selects the account with the lowest usage count. This maximizes your total available quota across all accounts.

**Best for**: Maximizing usage across multiple AWS Builder ID or IAM Identity Center accounts.

<Tip>
The `lowest-usage` strategy is the default and recommended for most users with multiple accounts.
</Tip>

## Adding Multiple Accounts

<Steps>
  <Step title="Authenticate with Kiro CLI">
    The recommended approach is to use `kiro-cli` to authenticate multiple accounts:

    ```bash
    kiro-cli login
    ```

    Follow the prompts to authenticate with AWS Builder ID or IAM Identity Center. Repeat this process for each account you want to add.
  </Step>

  <Step title="Enable Auto-Sync">
    Ensure auto-sync is enabled in `~/.config/opencode/kiro.json`:

    ```json
    {
      "auto_sync_kiro_cli": true
    }
    ```

    This automatically imports all authenticated sessions from your `kiro-cli` database.
  </Step>

  <Step title="Verify Accounts">
    The plugin will automatically detect and import all accounts from Kiro CLI. You can verify by checking the database:

    ```bash
    sqlite3 ~/.config/opencode/kiro.db "SELECT email, auth_method, is_healthy FROM accounts"
    ```
  </Step>
</Steps>

## Account Health Management

The plugin automatically tracks account health and removes unhealthy accounts from rotation.

### Account States

| State | Description | Action |
|-------|-------------|--------|
| **Healthy** | Account is active and available | Used for requests |
| **Rate-Limited** | Temporary quota exhaustion | Skipped until reset time |
| **Unhealthy** | Authentication or permission errors | Skipped with exponential backoff |
| **Permanently Failed** | Unrecoverable errors | Removed from rotation |

### Health Check Logic

From `src/core/account/account-selector.ts`, the plugin checks account health:

```typescript
const available = this.accounts.filter((a) => {
  if (!a.isHealthy) {
    if (isPermanentError(a.unhealthyReason)) {
      return false
    }
    if (a.failCount < 10 && a.recoveryTime && now >= a.recoveryTime) {
      a.isHealthy = true
      delete a.unhealthyReason
      delete a.recoveryTime
      return true
    }
    return false
  }
  return !(a.rateLimitResetTime && now < a.rateLimitResetTime)
})
```

<Warning>
Accounts with 10 or more consecutive failures are marked as permanently unhealthy and excluded from rotation.
</Warning>

## Usage Tracking

The plugin tracks usage metrics for each account to make intelligent rotation decisions.

### Automatic Usage Sync

From `src/core/account/usage-tracker.ts`, usage is synced after each successful request:

```typescript
async syncUsage(account: ManagedAccount, auth: KiroAuthDetails): Promise<void> {
  if (!this.config.usage_tracking_enabled) {
    return
  }
  
  const u = await fetchUsageLimits(auth)
  updateAccountQuota(account, u, this.accountManager)
  await this.repository.batchSave(this.accountManager.getAccounts())
}
```

### Usage Metrics

Each account tracks:
- **usedCount**: Number of requests made
- **limitCount**: Total quota available
- **lastUsed**: Timestamp of last request
- **rateLimitResetTime**: When rate limits will reset

## Deterministic Account IDs

Accounts are identified using a deterministic hash from `src/plugin/accounts.ts`:

```typescript
export function createDeterministicAccountId(
  email: string,
  method: string,
  clientId?: string,
  profileArn?: string
): string {
  return createHash('sha256')
    .update(`${email}:${method}:${clientId || ''}:${profileArn || ''}`)
    .digest('hex')
}
```

This ensures the same account credentials always map to the same ID, preventing duplicates.

## Circuit Breaker Protection

The plugin includes circuit breaker logic to prevent infinite retry loops:

```typescript
private checkCircuitBreaker(): void {
  if (Date.now() - this.lastCircuitBreakerReset > 60000) {
    this.circuitBreakerTrips = 0
    this.lastCircuitBreakerReset = Date.now()
  }

  if (this.circuitBreakerTrips >= 10) {
    throw new Error('Circuit breaker tripped: Too many consecutive failures')
  }
}
```

<Note>
If 10 consecutive account selection failures occur within 60 seconds, the circuit breaker trips and stops attempting requests.
</Note>

## Troubleshooting

### No Accounts Available

If you see "No accounts" errors:

1. Verify Kiro CLI authentication:
   ```bash
   kiro-cli login
   ```

2. Check auto-sync is enabled:
   ```bash
   grep auto_sync_kiro_cli ~/.config/opencode/kiro.json
   ```

3. Manually trigger sync by making a request after authentication

### All Accounts Rate-Limited

When all accounts hit rate limits:

- The plugin waits automatically for the shortest reset time
- You'll see a toast notification: "All accounts rate-limited. Waiting Xs..."
- Consider adding more accounts to increase total quota

### Account Health Issues

Check account status in the database:

```bash
sqlite3 ~/.config/opencode/kiro.db "SELECT email, is_healthy, unhealthy_reason, fail_count FROM accounts"
```

Accounts with `fail_count >= 10` are excluded from rotation.
